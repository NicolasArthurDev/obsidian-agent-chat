# SPEC 0003 — MCP connections

- **Status:** Accepted
- **Milestone:** M3
- **Date:** 2026-07-07

## Summary

Manage Model Context Protocol server connections and expose their tools to the agent.

## Motivation

MCP is the extensibility backbone: it lets the agent use external tools without bespoke
integrations.

## Requirements

- CRUD for MCP connections: create, edit, configure, remove, enable/disable.
- Two transports: local `stdio` (command + args + env) and remote HTTP/SSE (url + headers).
- Persist connection configs in settings.
- Connect on enable; disconnect on disable/unload; reconnect on demand; a "test connection"
  action.
- Discover each server's tools and register them in a `ToolRegistry`.
- The chat tool loop can invoke MCP tools and feed results back to the model.
- Non-functional: a failing server must not break the chat or other connections; errors are
  isolated per connection.

## Design

- `domain/mcp`: `McpServerConfig`, `McpConnection`, `McpTool`.
- `domain/tool`: `Tool`, `ToolRegistry` port.
- `infrastructure/mcp/McpClientAdapter`: wraps `@modelcontextprotocol/sdk` `Client` and the
  transports.
- `application/McpManager`: lifecycle + CRUD; `application/ToolExecutionService`: routing.
- `presentation`: MCP management UI in settings (list, add/edit form, test button).

## Out of scope

MCP prompts and resources (tools first); OAuth flows for remote servers (future).

## Open questions

- Where to source `stdio` binaries safely and how to warn about arbitrary command execution.
